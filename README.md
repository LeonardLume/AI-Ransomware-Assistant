# Ransomware Readiness AI Assistant

AI-assisted ransomware readiness interview for small and medium-sized organizations.

This project helps a user answer a structured readiness questionnaire in plain language, keeps the official score deterministic on the backend, and produces a backend-generated report with practical next steps.

## What This App Does

- asks a controlled interview from `data/questions.json`
- lets the user answer in free text
- uses AI to explain questions and interpret chat answers
- saves structured answers as `yes` / `partial` / `no` / `unsure`
- calculates the official score only from backend rules in `data/scoring_rules.json`
- generates a report, action plan, findings, and evidence checklist on the backend

## What It Does Not Do

- not a full security audit
- does not scan infrastructure
- does not pentest or enumerate anything external
- does not let the LLM invent the score
- does not provide offensive security, malware, bypass, or exploit guidance

## How It Works

The system has three important layers:

1. `questions.json` is the interview source of truth.
2. `scoring_rules.json` is the scoring source of truth.
3. the AI chat helps with conversation, clarification, and free-text interpretation, but it does not control the score.

Current behavior:

- the chat is AI-led when a real provider is configured
- the backend still validates extracted answers before saving them
- the report text is generated by the backend, not by the LLM
- defensive `skills/` files improve explanations, action plans, and evidence guidance

## Main Stack

- `backend/`: FastAPI API and interview logic
- `frontend-web/`: React + Vite web app
- `data/`: questions, scoring rules, domain metadata, demo profiles
- `skills/`: defensive playbooks used for explanations and recommendations
- `docker-compose.yml`: production/demo deployment with Caddy + backend

There is also an older Streamlit UI in `frontend/`, but the main product UI is the React app in `frontend-web/`.

## Choose One Start Method

For most users there are only 2 recommended ways to run the project:

1. local development/demo on your machine
2. Docker deployment

Everything else is advanced/manual.

## Local Start

Requirements:

- Python 3.10+
- Node.js 18+
- npm

### Windows

Recommended:

```powershell
.\start.bat
```

This starts:

- FastAPI backend
- React/Vite frontend
- automatic dependency install when needed

Open:

- frontend: `http://localhost:5173`
- backend docs: `http://127.0.0.1:8000/docs`
- provider status: `http://127.0.0.1:8000/provider/status`

### macOS / Linux

```bash
chmod +x start.sh scripts/dev.sh
./start.sh
```

## Public Demo Link

If you want a temporary public demo link from your local machine:

### Windows

```powershell
.\start_public.bat
```

### macOS / Linux

```bash
./start_public.sh
```

This uses a Cloudflare Quick Tunnel and exposes the frontend with `/api` proxied back to the local backend.

## Docker Start

Requirements:

- Docker Desktop on Windows, or Docker Engine on Linux/macOS

### Start

Windows:

```powershell
copy .env.production.example .env
docker compose up -d --build
```

macOS / Linux:

```bash
cp .env.production.example .env
docker compose up -d --build
```

Open:

```text
http://localhost
```

In Docker mode:

- Caddy serves the built React frontend
- `/api/*` is proxied to FastAPI
- the backend is not exposed directly on a host port by default

Useful Docker commands:

```bash
docker compose logs -f
docker compose down
docker compose up -d --build
```

Important: after backend or prompt changes, rebuild the containers with `docker compose up -d --build`. Restarting only the browser is not enough.

## About the BAT Files

There are several helper scripts in the repo, but you can ignore most of them.

Recommended scripts:

- `start.bat`: normal local full-stack start
- `start_public.bat`: local full-stack start with temporary public tunnel

Advanced/manual scripts:

- `run_backend.bat`: backend only
- `run_frontend.bat`: frontend only
- `scripts/dev.ps1`: source-of-truth PowerShell launcher used by the wrappers

If you just want to use the project, start with `start.bat` or Docker. You do not need the other batch files.

## LLM Provider Setup

Create `.env` from `.env.example` for local runs.

### OpenAI-compatible Provider

```env
LLM_PROVIDER=openai
OPENAI_API_KEY=your-api-key-here
OPENAI_MODEL=gpt-4o-mini
OPENAI_BASE_URL=https://api.openai.com/v1
REQUEST_TIMEOUT_SECONDS=120
```

OpenRouter also works:

```env
LLM_PROVIDER=openai
OPENAI_API_KEY=your-openrouter-key-here
OPENAI_MODEL=openai/gpt-4o-mini
OPENAI_BASE_URL=https://openrouter.ai/api/v1
REQUEST_TIMEOUT_SECONDS=120
```

### Ollama

```env
LLM_PROVIDER=ollama
OLLAMA_MODEL=qwen2.5:7b
OLLAMA_URL=http://localhost:11434/api/generate
REQUEST_TIMEOUT_SECONDS=120
```

### Fallback Mode

```env
LLM_PROVIDER=fallback
REQUEST_TIMEOUT_SECONDS=120
```

Fallback mode keeps the demo usable, but the chat is less natural than with a real provider.

## Security and Runtime Notes

Optional hardening env vars:

- `API_AUTH_TOKEN`: bearer-token protection for non-public API routes
- `RATE_LIMIT_CHAT_PER_MINUTE`
- `RATE_LIMIT_REPORT_PER_MINUTE`
- `RATE_LIMIT_DEMO_PER_MINUTE`
- `TRUST_PROXY_HEADERS=1`

Docker sessions are stored in the `backend_data` volume at `/app/data_runtime`. If you remove the volume, the saved sessions are lost.

## User-Facing Flow

Typical flow:

1. open the app
2. start the interview
3. answer in normal language
4. ask clarification questions when needed
5. finish the interview
6. read the readiness report and next steps

## Project Notes for Developers

- official score is always backend-controlled
- report narrative is backend-generated
- AI chat can explain and interpret, but backend validation decides what gets saved
- `skills/` improve guidance and evidence recommendations, not the numeric score
- the current main UI is React; Streamlit is legacy/fallback

## Limits

- depends on user-provided answers
- not a substitute for a real audit or incident response retainer
- fallback mode is useful for demos, but real LLM mode gives a much better chat experience
